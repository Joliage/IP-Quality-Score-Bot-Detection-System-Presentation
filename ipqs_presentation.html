Project Context

DSpace 7 & SOLR Statistics

03 / 30

What is DSpace?

  • DSpace 7 is an open-source repository system used by UNESCAP to host research publications and documents.
  • SOLR (Apache Solr) powers the search and statistics engine — it logs every page view and download with IP, user agent, timestamp, and document IDs.
  • These statistics are critical for measuring the impact and reach of publications.

The Bot Problem

  • Bots (Googlebot, Bingbot, scrapers) generate fake page views and downloads that inflate statistics.
  • The editing model changes have worsened bot contamination of SOLR data.
  • We need to filter out bot traffic and keep only legitimate human interactions.
  • Solution: Use IP Quality Score APIs to classify each IP as bot or human.

Security Indicators

The 6 IPQS Flags

05 / 30
🔒

is_vpn

IP belongs to a known VPN provider. Medium risk alone, high when combined with other flags.

🌐

is_proxy

IP operates as an open/anonymous proxy (HTTP/SOCKS5). 90%+ of malicious bots use proxies. Very high risk.

🧅

is_tor

IP is a Tor exit node. Nearly all Tor traffic in analytics should be filtered. Very high risk.

🖥️

is_hosting

IP belongs to cloud/VPS/datacenter. Mobile User-Agent from hosting IP = near-certain bot.

⚠️

is_abuse

IP reported for abuse in last 30 days (spam, scraping, brute-force). Direct evidence of malicious automation.

📨

is_relay

IP belongs to email relay, CDN edge, or traffic forwarding service. Context-dependent risk.

Flag Details

VPN & Proxy Detection

06 / 30

is_vpn — Virtual Private Network

  • Detection: ASN/IP range ownership analysis, provider database matching, traffic pattern profiling
  • Legitimate: Privacy protection, remote work, bypassing geo-restrictions
  • Malicious: Hiding botnet origin, evading IP bans, simulating users from multiple countries
  • Risk: Medium alone → High in context (with non-residential connection_type + high fraud_score)

is_proxy — Anonymous Proxy Server

  • Detection: Active port scanning (80, 8080, 3128), known proxy databases, historical forwarding behavior
  • Types: Public proxies, elite proxies, datacenter proxies
  • Malicious: Botnets for scraping, CAPTCHA solving farms, credential stuffing
  • Risk: HIGH — 90%+ of malicious bots use proxies. Triggers immediate suspicion even with low fraud_score.

Flag Details

Tor & Hosting Detection

07 / 30

is_tor — Tor Exit Node

  • Detection: Real-time sync with Tor Project's official exit node list
  • Legitimate: Anonymity for activists, journalists, whistleblowers
  • Malicious: Automated abuse, dark pool scraping, DDoS coordination
  • Risk: VERY HIGH — block by default in analytics. is_tor=true → filter immediately.

is_hosting — Datacenter/Cloud IP

  • Detection: ASN classification, BGP routing data, reverse DNS patterns (e.g. ec2-...amazonaws.com)
  • Legitimate: Web servers, APIs, CDNs, enterprise tools
  • Malicious: Bot farms on cheap VPS, scraping clusters, fraud rings using cloud IPs
  • Key insight: A mobile User-Agent from a hosting IP = near-certain bot

Flag Details

Abuse & Relay Detection

08 / 30

is_abuse — Recent Abuse Reports

  • Detection: IPQS Fraud Network (community + partner reports), honeypot traps, partner API feedback loop
  • Severity: Tied to abuse_velocity: low, medium, high
  • Risk: HIGH — direct evidence of malicious automation
  • Example: IP with 47 scraping reports in 24h → is_abuse=true, abuse_velocity=high → Active bot

is_relay — Email/Traffic Relay

  • Detection: Known relay ASN/IP ranges, header analysis, service fingerprinting
  • Legitimate: Email delivery, CDN caching, Apple iCloud Private Relay
  • Malicious: Abusing free tiers for bot traffic, hiding origin, spam relay chains
  • Risk: Context-dependent. Apple Private Relay (is_relay=true) increasingly used by real iOS users.

Risk Scoring

How the Score Is Calculated

09 / 30

20 Points Per Flag

flags = [is_vpn, is_proxy, is_tor, is_hosting, is_abuse, is_relay] risk_score = sum(20 for f in flags if f) # Range: 0 (clean) to 120 (max risk) is_bot = any(flags) # True if ANY flag is True classification = "BOT DETECTED" if is_bot else "NOT A BOT"

Why 20 Points Per Flag?

  • Simplicity — easy to understand and audit
  • Equal weighting — each anonymity/abuse signal is considered equally dangerous
  • Clear scale — 0 = no risk, 120 = maximum risk
  • Any single flag triggers bot classification

Example

IP 66.249.80.196: is_hosting=True + is_relay=True → score = 40/120 → BOT DETECTED

API Architecture

Dual-API Strategy

10 / 30

Primary: AbstractAPI IP-Intelligence

  • Returns the 6 security flags (VPN, proxy, Tor, hosting, abuse, relay)
  • Company, ASN, geolocation, timezone data
  • API key authentication, 10-second timeout
  • Rate-limited: 1.1s delay between requests

Fallback: ip-api.com

  • Lightweight geo/ISP data: country, city, ISP, org
  • Free tier, no API key required
  • Used to fill missing fields from primary source
  • Fields requested: status, country, regionName, city, isp, org, proxy, hosting

Architecture

System Pipeline

11 / 30
1. IngestLoad SOLR JSON data (pageviews & downloads) from DSpace 7
2. ExtractDeduplicate IPs from raw records (set-based extraction)
3. ClassifyQuery IPQS API for each unique IP → 6 flags → risk score
4. StoreSave bot_mapping to MongoDB. Insert raw_data. Filter clean data.
5. Outputfiltered_data collection contains only human traffic.

Key Optimization

With 5,000 free API lookups, we extract unique IPs first (not parse raw data). 1,000 records may have only ~200 unique IPs. The bot_mapping collection caches results so we never re-query the same IP twice.

MongoDB Design

Database: solr_statistics_dbS

12 / 30

raw_data

All original SOLR documents — both bot and human traffic. Inserted via insert_raw_data.py.

Fields: ip, dns, userAgent, isBot, type, owningItem, time, statistics_type, uid

bot_mapping

Each unique IP → is_bot boolean + full API response. Created by bot_mapping.py.

Fields: ip, is_bot, response (full IPQS data). IPs deduplicated via raw_data.distinct("ip") — no re-queries on existing IPs.

filtered_data

Only records where the IP is NOT a bot (determined by bot_mapping, no re-calling API). Created by filtered_data.py.

Contains clean human traffic only — ready for accurate statistics reporting.

SOLR Statistics JSON Structure

14 / 30

One SOLR Document

{ "ip": "66.249.80.196", "dns": "google-proxy-66-249-80-196", "userAgent": "Mozilla/5.0 (compatible; Google AppsViewer; ...)", "isBot": true, "type": 0, "owningItem": ["8f44d336-f51f-..."], "owningColl": ["e5a652c3-655e-..."], "time": "2025-02-04T15:38:38.514Z", "bundleName": ["ORIGINAL"], "statistics_type": "view", "uid": "5c40ac51-437b-..." }

Key Fields

  • ip: The visitor's IP address — the target for bot detection
  • dns: Reverse DNS lookup — often reveals bot identity (e.g. "google-proxy-...")
  • userAgent: Browser/bot identifier — Googlebot, Bingbot, etc.
  • isBot: SOLR's own guess (unreliable — we verify with IPQS)
  • statistics_type: "view" or "download" — the event type
  • owningItem: UUID of the publication being accessed
  • time: ISO 8601 timestamp of the event

insert_raw_data.py — Loading Data

20 / 30
from pymongo import MongoClient import json client = MongoClient("mongodb://localhost:27017/") db = client["solr_statistics_dbS"] raw_data_col = db["raw_data"] with open("data_may_01_limit_1000.json") as f: data = json.load(f) docs = data["response"]["docs"] raw_data_col.insert_many(docs) print(f"Inserted {len(docs)} documents")

What This Does

  • Connects to local MongoDB instance on default port 27017
  • Creates/connects to database solr_statistics_dbS
  • Loads the full SOLR JSON export (1000 records)
  • Extracts the docs array from response.docs
  • Uses insert_many() for bulk insertion — much faster than one-by-one
  • Prints confirmation with document count

bot_mapping.py — IP Classification

21 / 30

Key Logic (Updated)

# STEP 1: Get unique IPs from MongoDB unique_ips = raw_data.distinct("ip") existing = set(bot_mapping.distinct("ip")) ips_to_lookup = [ip for ip in unique_ips if ip not in existing] # Enforce 5000 free-tier limit if len(ips_to_lookup) > LOOKUP_LIMIT: ips_to_lookup = ips_to_lookup[:LOOKUP_LIMIT] # STEP 2: Rate-limited lookup loop for ip in tqdm(ips_to_lookup): r = requests.get(API_URL, params={...}) bot_flag = is_ip_bot(r.json()) bot_mapping.insert_one({ "ip": ip, "is_bot": bot_flag, "response": data }) time.sleep(1.1) # STEP 3: Build clean_data directly map_dict = {m["ip"]: m["is_bot"] for m in bot_mapping.find({})} to_insert = [doc for doc in raw_data.find() if map_dict.get(doc["ip"]) is False] clean_data.insert_many(to_insert)

Smart Design Choices

  • MongoDB-driven: IPs come from raw_data.distinct("ip") — no need to re-load JSON
  • Skip existing: IPs already in bot_mapping are not re-queried (saves API calls on re-runs)
  • 5000 cap: Respects the IPQS free-tier API limit
  • 7 security flags: is_vpn, is_proxy, is_tor, is_hosting, is_relay, is_mobile, is_abuse
  • Rate limiting: 1.1-second time.sleep() between calls keeps free-tier safe
  • Three-step pipeline: extract IPs → query API → build clean_data in one script
  • Error handling: On API failure, marks IP as bot (safe default) with error stored in response
  • tqdm progress bar: Visual feedback during long-running lookups

filtered_data.py & Filtering.py

filtered_data.py — Simple Filter

clean = db["clean_data"] filtered = db["filtered_data"] non_bot = clean.find({"isBot": False}) count = 0 for doc in non_bot: filtered.insert_one(doc) count += 1 print(f"Inserted {count} non-bot docs")

Reads clean_data, keeps only isBot=False records, inserts into filtered_data collection.

Filtering.py — Advanced Pipeline

  • Loads bot_mapping into memory as a dict for O(1) lookups
  • Filters bot traffic: skips any IP where bot_map[ip] is True
  • Fetches metadata: calls ESCAP repository API to get publication title, handle, program of work
  • Aggregates monthly stats: groups views by (pub_id, year, month)
  • Stores in MongoDB: publications collection + publication_stats_monthly with upsert/$inc
  • Two additional collections: publications and publication_stats_monthly

Error Handling

Robustness & Edge Cases

26 / 30

API Failures

If AbstractAPI times out or returns an error, the IP is marked as is_bot=True (safe default). Error string is stored in the response field for debugging.

Rate Limiting

1.1-second delay between API calls. Elapsed time tracking ensures we never exceed the free-tier rate limit. tqdm shows progress during long-running lookups.

Deduplication

MongoDB distinct("ip") guarantees IP uniqueness from raw_data. The skip-existing check against bot_mapping ensures we never re-query an IP we've already classified.
